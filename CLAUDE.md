alembic upgrade head                         # apply migrations
alembic revision --autogenerate -m "desc"    # create migration

Docker (full stack):
```bash
docker compose up -d --build      # build + start postgres, redis, backend, frontend

docker compose logs -f backend    # follow backend logs
curl http://localhost:8000/health # backend smoke

```

## Architecture (implemented)

### Backend layout

- `app/main.py` â€” `create_app()` factory; `GET /health` checks DB + Redis connectivity.
- `app/core/config.py` â€” `Settings(BaseSettings)`; reads `DATABASE_URL`, `REDIS_URL`, JWT settings from env.
- `app/db/base.py` â€” `Base(DeclarativeBase)` plus reusable mixins: `UUIDPrimaryKey` (`gen_random_uuid()` server default), `TimestampMixin`, `SoftDeleteMixin` (`deleted_at`), `TenantScopedMixin` (`organization_id` FK, NOT NULL on content tables), `AuditSubjectMixin` (`created_by_id`/`updated_by_id`/`reviewed_by_id`).
- `app/db/queries.py` â€” `not_deleted(model)` helper (filters `deleted_at IS NULL`). Use this for soft-delete queries.
- `app/db/seed.py` â€” idempotent seed guarded by `SchemaMeta.seed_version`. `run_seed(session)` + `main()` CLI.
- `app/db/session.py` â€” engine/sessionmaker/session factory.
- `app/models/` â€” one module per bounded context: `auth`, `taxonomy`, `question`, `practice`, `exam`, `admin`, plus `enums.py`. All registered via `app/models/__init__.py`.
- `app/services/snapshot.py` â€” `snapshot_question(question, options) -> dict` for historical-integrity snapshots.
- `app/services/audit.py` â€” `log_audit(session, *, action, ...)` helper.
- `app/alembic/` â€” `env.py` respects an explicitly-set `sqlalchemy.url` (so tests can target a different DB); `versions/66bec070d8fc_initial_schema.py` is the initial migration.

### Cross-cutting rules (carry into later sub-projects)

- **Service-layer backend**: API routes delegate to service modules that own business logic and DB access. Never put business logic directly in route handlers.
- **Tenant scoping**: content tables (questions, books, chapters, practice/exam sessions, import jobs, etc.) are `organization_id`-scoped and NOT NULL. Taxonomy (`ExamBlueprint`, `ExamDomain`, `KnowledgePoint`, `KnowledgePointDomain`, `Tag`) is GLOBAL (shared across orgs).
- **Native PostgreSQL ENUMs**: defined in `app/models/enums.py` and created as real `CREATE TYPE` in the migration. The initial migration's `downgrade()` explicitly drops these types (autogen `drop_table` does not).
- **UUID PKs**: all PKs are UUID with `gen_random_uuid()` server default. Email uniqueness is case-insensitive via a hand-written functional index `uq_users_email_lower` (not expressible in column metadata).
- **Historical integrity via snapshots**: completed practice/exam answers store a snapshot of the question and options at answer time (`PracticeAnswer`, `ExamAnswer` JSONB columns), so later edits to a question never change past records. Use `snapshot_question()`.
- **Soft delete only**: deleting a question must not break historical answer records. Filter live rows with `not_deleted(model)`.
- **Exam config is data, not code**: domain weights, question-count ranges, exam duration, passing line, and effective dates live in `ExamBlueprint`/`ExamDomain`. Do not hardcode CISSP domain weights or the 100â€“150 / 3-hour / 700-pass rules.
- **Audit logging**: logins, imports, edits, publishes, deletes, and permission changes go to `AuditLog` via `log_audit()`.
- **CAT is a study tool, not an official prediction** (PRD Â§11): the MVP CAT must be rule-driven with simplified ability estimation. Do not make 3PL IRT a P0 dependency â€” full IRT is a Phase 5 enhancement. `backend/app/services/cat_engine.py` implements the rule-driven MVP engine (pure module, no SQLAlchemy); CAT reports MUST surface `cat_engine.DISCLAIMER` (â‰  ISC2 official scoring â€” FR-CAT-10).
- **CAT exam rules** (when implemented): once submitted, an answer cannot be revised; no skipping; forward-only; medium-difficulty start item.

### Frontend layout

- `src/app/layout.tsx` â€” root layout; loads DM Sans via `next/font/google` (`variable: "--font-sans"`) and applies `font-sans antialiased` to `<body>`.
- `src/app/globals.css` â€” the design-token source of truth. `:root` defines HSL-triplet semantic roles (`--primary`, `--card`, `--border`, `--muted-foreground`, `--destructive`, `--success`, `--ring`, â€¦) consumed as `hsl(var(--*))` throughout. Also defines `--radius`/`--radius-lg`, `--canvas` (app bg), `--hero-*` (auth gradient), and a `--shadow-*` ramp. **Light mode only** â€” there is no `.dark` block, no `next-themes`. Change colors here, not in components.
- `tailwind.config.ts` â€” maps the CSS vars to Tailwind: `colors.*` â†’ `hsl(var(--*))`, `fontFamily.sans` â†’ `var(--font-sans)` first, `borderRadius.lg2` â†’ `--radius-lg`, `boxShadow.{card,float,sm,xl}` â†’ the shadow vars. `content` globs `app/`, `components/`, `features/`.
- `src/app/(app)/layout.tsx` â€” authenticated shell: `RequireAuth` + `AppSidebar` + `<main>` on `bg-canvas`. `src/app/(auth)/layout.tsx` â€” centered card on the `--hero-*` vertical gradient.
- `src/components/app-sidebar.tsx` â€” client component owning nav, the permission-gated "Manage" group, a **Settings link** (the old in-sidebar language `<Select>` was relocated to `/settings`), and logout. Nav items use Apple pill styling (active = `bg-accent text-foreground`). All labels route through `t()`.
- `src/lib/i18n/` â€” the UI-i18n infrastructure (PRD v1.2): `types.ts` (`Locale = "en"|"zh"`, `TFn`), `provider.tsx` (`<I18nProvider initialLocale>` + `useI18n`/`useT`), `cookie.ts` (`ui_lang` cookie read/write). The root `app/layout.tsx` reads the cookie server-side â†’ `<html lang>` + seeds `initialLocale` into `<Providers>` (no English flash). `src/locales/{en,zh}.ts` are the single source of UI-chrome strings (zh is typed `typeof en`; parity enforced by `locales/__tests__/i18n.test.ts`); `src/locales/t.ts` is `makeT(dict)` (dot-path resolve + `{var}` interpolation, falls back to the key when missing). `src/features/shared/enum-label.ts` provides `enumLabel(t, scope, key)` for enum badges. Component tests render via `src/test/render-with-providers.tsx` (`I18nProvider` + QueryClient, `initialLocale` defaults `"en"` so English assertions still hold).
- `src/components/ui/*` â€” shadcn primitives (button, card, input, select, tabs, dialog, â€¦). `Button` has a `size: "pill"` variant + `font-semibold` base; `Card` defaults to `shadow-card` with an opt-in `hover` lift. Prefer these over hand-rolled markup.
- `src/components/{eyebrow,field,page-header,bilingual-text}.tsx` â€” small shared primitives. `<Eyebrow>` = section label; `<Field>` = icon-leading input wrapper (pass `<Input>` as children so a11y stays on the real input); `<PageHeader>` has an `eyebrow` prop; `<BilingualText>` renders en/zh/both by `mode`.
- `src/features/<context>/` â€” feature modules hold the real page logic (e.g. `features/exam/cat-runner.tsx`, `features/practice/runner-machine.ts`). **Pages in `src/app/.../page.tsx` are thin wrappers** â€” when editing a page's behavior or styling, look in the corresponding `features/` module, not just `page.tsx`. All feature modules route visible chrome through `useT()`; pure label helpers (`features/analytics/format.ts`, `features/exam/format.ts`, `features/questions/labels.ts`) accept a `TFn` so enum labels translate. Several feature modules have unit tests (`features/**/__tests__/`) that define their contracts (e.g. `cat-runner.test.tsx` asserts the language toggle never advances the item; `editor.test.tsx` asserts FR-LANG-09 publish-completeness validation) â€” preserve these when restyling.

### Cross-cutting frontend rules

- **Language toggle is pure client state.** In practice/exam runners, switching en/zh/bilingual (question-content `language_mode`) must only update local `mode` state â€” it must **never** call `/next` or any advance/submit function (this is the CAT forward-only invariant, enforced by `cat-runner.test.tsx`). Delivery endpoints already return BOTH languages so no refetch is needed.
- **Interface language (UI chrome) is separate from question-content language.** `interface_language` (en/zh) drives `t()` / `<I18nProvider>`; `language_mode` (en/zh/bilingual) drives `<BilingualText>` question rendering. They never affect each other. Switching interface language updates only client `locale` state + writes the `ui_lang` cookie + PUTs `/api/users/me/preferences` â€” like the content toggle, it must never advance/submit. All hand-written UI chrome MUST flow through `t()`; **never translate taxonomy data** (domain/book/chapter/knowledge-point/tag names) or question content (FR-I18N-05). When adding a UI string, add the key to BOTH `locales/en.ts` and `locales/zh.ts` (the parity test enforces this). Component tests must render via `renderWithProviders` (default `initialLocale="en"`), so English-text assertions still resolve.
- **Preserve tested contracts when restyling.** Feature-module tests target accessible names, label associations, and state-machine behavior â€” when changing JSX/className, keep `id`/`htmlFor`/`aria-*`/button text/`role` stable so those queries still resolve.
- **No dark mode.** Do not wire `next-themes`, `.dark` tokens, or theme toggles without a separate scope decision.

### Tests

- Tests use a real PostgreSQL DB, not SQLite. `tests/conftest.py` creates/drops a dedicated `cissp_test` database each session (isolated from the dev `cissp` DB) and uses per-test transaction rollback via nested SAVEPOINT.
- `tests/test_migrations.py` runs against a separate `cissp_migtest` DB (drop/create/upgrade per fixture) and includes a no-autogenerate-drift test â€” the highest-value guard. When you change models, run `alembic revision --autogenerate` and keep drift at zero (the test filters out the hand-written `uq_users_email_lower` index and throwaway `_test_*` tables).
